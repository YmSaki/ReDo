# 今後の拡張計画

## ESLint プラグイン (別プロジェクト)

### パッケージ名
`eslint-plugin-redo`

### 実装する機能

#### ③ key 戦略の明文化
- 配列の `.map()` などで要素を生成する際の key 属性ルールを明文化
- Reactの `jsx-key` ルールと同様のアプローチ

#### ④ key warn の実装
- key が必要な場所で key がない場合に警告を出す
- リスト要素に key がない場合の警告
- key の値が適切でない場合の警告（例: index を key にしている場合）

### 実装方針
- ESLint プラグインとして独立したパッケージで実装
- ReDo の JSX パターンを検出してルールを適用
- 開発時にリアルタイムでフィードバックを提供

### なぜ別プロジェクトか
- コアフレームワークとは責務が異なる（静的解析 vs ランタイム）
- ESLint プラグインのセットアップや依存関係が独立している
- オプショナルな開発ツールとして提供する方が適切

## 「他人を指す」アドレッシング（v1では見送り。Issue #8関連）

Issue #8 で「コントローラ + this.reRender()」パターンを実装した際、対になる発想として
「発火要素（トリガー）から、それが属する島を逆引きする」アドレッシングの是非を検討した。
具体的には以下の2つのAPIが候補に挙がった。

### `ctx.getComponent(element)`
発火要素（例: `ctx.payload` がDOMイベントならその `.target`）から、その要素が属する島を
逆引きするAPI。DOMには island.ts の `mountIsland()` が既に `(dom as any).__redoIsland` を
スタンプしているため（`src/redo/island.ts` 参照）、実装自体はDOMツリーを親方向に辿って
`__redoIsland` を持つ要素を探すだけで技術的には可能。

### `getComponentByQuery(selector)`
例えば `.popup Button` のようなセレクタでツリーを辿り、該当する島（や島の中の特定要素）を
探し当てるAPI。上記の逆引きをさらに一般化したもの。

### なぜv1で見送るか
- **「自分」を再描画するのに他人へのアドレッシングは不要**: コントローラは自分がどのView
  （どの島）に属しているかを常に知っている（`this.View`）。`this.reRender()` = 
  `reRenderIsland(this.View)` で完結するため、「発火要素→島」の逆引きが必要になる場面は
  「自分の再描画」には現れない。
- **決定的に効かないケースがある**: 非同期完了時（`ctx.run(task, payload, { success: this.apply })`
  の `apply`）では、`ctx.payload` は非同期処理の結果であってDOM要素ではない。この場合
  `ctx.getComponent(element)` は要素を渡すこと自体ができず無力になる。一方 `this.reRender()`
  はインスタンスが `this.View` への参照を直接握っているため、この場合でも確実に動く。
  つまり「他人を指すアドレッシング」は `this.reRender()` の完全な代替にはなり得ず、
  カバーできる範囲が真に狭いサブセットになる。
- **用途は「他人の島を外から叩く」ケースに限られる**: 例えばポップアップの外側にある
  ボタンから、ポップアップ内の別コンポーネントを再描画したい、といったケース。これは
  v1では「親から子に手を伸ばす」設計（props自動伝播や、コントローラ間の明示的な参照受け渡し）
  で代替でき、専用APIを今すぐ用意する必要性が薄い。
- **セレクタベースの逆引き（`getComponentByQuery`）は輪をかけて薄い基盤の上に立つ**:
  DOM構造への依存が強く（マークアップを変えるとクエリが壊れる）、`__redoIsland` の
  スタンプは「島のルート」にしか付かないため、`.popup Button` のような「島の中の任意の
  要素」を狙うクエリは、島の内部実装（v1では不透明・親から立ち入り禁止）にまで踏み込む
  ことになり、island.ts の「境界の内部はその島だけが所有する」という前提と衝突しやすい。

### `getComponent(this)` を作らない理由
コントローラのメソッド内で `this` と書けている時点で、すでにそのコントローラの
インスタンス（＝島の所有者）の上に立っている。そこから改めて「自分が属する島は何か」を
逆引きする `getComponent(this)` を用意すると、`this` → 逆引き → 結局 `this.View` に戻る、
という循環になり、`this.reRender()` に対して何の価値も足さない。したがって
`getComponent(this)` のような自己参照専用のヘルパーは作らない。

### 再検討する条件（将来）
「他人の島を外から明示的に再描画したい」という具体的なユースケース（例: グローバルな
通知センターが、任意の場所に浮かぶポップアップ島を閉じる、等）が繰り返し必要になった時点で、
`ctx.getComponent(element)` のような限定的な逆引きAPIを、DOM要素を渡せる同期イベント
経路に限定して（非同期完了時には無力である前提を明記した上で）再検討する。
